# Локальный исследовательский инструмент (ComfyUI) для подбора оптимальной конфигурации удаления водяного знака

Цель: сделать **локальный, понятный заказчику инструмент‑исследователь**, который даёт:
- оценку **ресурсов на одно изображение** (время, VRAM, RAM, CPU, диск),
- **ожидаемое качество** результата (риск артефактов/недоудаления),
- **прозрачный выбор оптимальной опции** под конкретный кейс.

Инструмент не заменяет сам пайплайн ComfyUI, а **упаковывает его в “понятный калькулятор”** с удобной конфигурацией и отчётом.

---

## 1) Что получает заказчик

**Понятный отчёт** с 2–4 вариантами обработки и честными компромиссами:
- **Вариант A — Быстро** (минимум времени/стоимости, достаточное качество)
- **Вариант B — Сбалансировано** (оптимум цена/качество)
- **Вариант C — Качество** (максимально чисто, но дольше/дороже)
- **Вариант D — Risk‑safe** (с доп‑валидацией и повторным прогоном сложных кадров)

Каждый вариант показывает:
- время/изображение (median и p95),
- VRAM пик, RAM пик, CPU загрузку,
- долю “сомнительных” кадров,
- прогноз времени на весь объём (например 25k).

---

## 2) Концепция инструмента (кратко)

**Explorer** = локальное приложение (CLI + Web UI), которое:
1) Берёт **подвыборку** изображений (например 100–500)
2) Прогоняет 2–4 заранее подготовленных **ComfyUI‑пресета**
3) Меряет **ресурсы и качество**
4) Формирует **отчёт и рекомендации**

## 4) Комфортная переконфигурация (UX)

### 4.1 Быстрые пресеты (для заказчика)
- `Fast` — минимум шагов, лёгкая модель, минимальный контроль
- `Balanced` — разумное соотношение
- `Quality` — больше шагов, tile‑inpaint, усиленная проверка
- `Safe` — повторная обработка “плохих” кадров

### 4.2 Продвинутые настройки (для оператора)
- Шаги, CFG, sampler
- Тип квантизации (fp16 / q5 / q4)
- Размер батча
- Tile‑inpaint (да/нет)
- Маска: пороги, дилатация, фильтрация
- Режим валидации качества

### 4.3 Принцип “1 экран — 1 решение”
UI показывает **2–4 карточки вариантов**. Заказчик сравнивает и выбирает.

## 5) Опциональные функции (включаются по флагу)

| Опция | Что даёт | Когда включать |
|------|----------|----------------|
| Quality Check | Авто‑поиск артефактов и остатка watermark | Важное качество |
| Re‑run Bad Frames | Автоматический прогон только проблемных | Большие батчи |
| Tile‑Inpaint | Лучшая работа на больших изображениях | >1024px |
| Multi‑GPU | Параллельная обработка | 2+ GPU |
| Cloud Cost Estimator | Прогноз стоимости при переносе в облако | Для масштабирования |
| Mask Debug Mode | Сохранение масок для проверки | Во время настройки |

## 6) Метрики и как считаем “качество”

### 6.1 Метрики ресурсов
- **time/img**: p50, p95
- **VRAM peak**: max GPU usage
- **RAM peak**, **CPU avg**
- **Disk IO**: чтение/запись на батч

### 6.2 Метрики качества (комбинированный скоринг)
- **Residual Watermark Score** (детектор текста/серых областей)
- **SSIM вне маски** (не должен падать)
- **Edge/Detail Loss** (детали не смазываются)
- **Manual Spot‑check** (например 20 кадров)

Финальный скоринг:
```
quality_score = 0.4 * (1 - residual_wm)
               + 0.3 * ssim_outside
               + 0.2 * edge_preserve
               + 0.1 * manual_ok_rate
```

## 7) Методология эксперимента (как оцениваем)

1) **Отбор подвыборки** (N=100–500, равномерно по типам изображений)
2) **Warm‑up** 5–10 кадров (прогрев модели)
3) **Запуск 2–4 пресетов**
4) **Сбор статистики** и вычисление p50/p95
5) **Формирование прогноза** для полного объёма (например 25k)

Формула прогноза:
```
Total_time ≈ (p50_time * 0.7 + p95_time * 0.3) * total_images
```

## 9) Техническая реализация (локально)

### 9.1 Компоненты
- **ComfyUI**: основной workflow
- **Explorer Runner** (Python): запускает пайплайн и логирует
- **Metrics Agent**: читает GPU/CPU/RAM (nvidia‑smi + psutil)
- **Quality Agent**: считает SSIM / residual wm
- **UI**: Streamlit / FastAPI + React (локально)

## 10) План работ (детальный breakdown)

### Этап 1 — Подготовка пайплайна (1–2 дня)
- собрать 2–4 ComfyUI‑пресета
- стабилизировать маску водяного знака
- подтвердить детерминизм

### Этап 2 — Инструмент‑Runner (2–3 дня)
- запуск workflow через API
- батчи + логирование времени
- запись метрик на батч и на кадр

### Этап 3 — Оценка качества (2–3 дня)
- residual watermark detector
- SSIM вне маски
- сбор manual spot‑check

### Этап 4 — UI и отчёты (2 дня)
- экран выбора пресета
- запуск/статус/прогресс
- итоговый отчёт с таблицами

### Этап 5 — Финализация и демонстрация (1 день)
- финальный прогон
- презентация результатов заказчику

---

## 11) Риски и как их закрыть

- **Слишком разные типы watermark** → добавить 2 режима маски
- **Сложные кадры** → Safe preset с повторной обработкой
- **Просадки качества** → ручная проверка + авто‑флаги
- **Пиковая память** → уменьшение батча / q4

---

## 12) Результат

В итоге заказчик получает **понятный локальный “калькулятор качества и ресурсов”**, где можно за 20–30 минут понять:
- **сколько займёт обработка всего объёма**,
- **какой режим даст нужное качество**,
- **какой режим наиболее выгоден по затратам**.

Инструмент масштабируется: сначала локально, потом легко переносится в облако, потому что все замеры и пресеты уже формализованы.
